# Code Review Handoff

## 适用场景

当你遇到以下情况时，可按需使用这个 skill：

- 你已经完成本轮实现，准备输出 `implementation_result`
- 你需要把改动重点、影响范围和风险点交接给 `Tester` 或最终人工审阅方
- 你希望在提交结果前做一次简短自审，避免遗漏明显问题
- 这轮改动不大，但涉及关键路径、状态切换或易回归区域

## 何时不要用

以下情况通常不需要使用这个 skill：

- 当前没有发生实现改动，只是在调研或规划
- 你还没完成实现，连本轮实际改动面都不清楚
- 下游并不需要接收任何关于实现的交接信息

## 推荐流程

1. 先概括本轮到底改了什么，不要把无关背景写成改动点。
2. 列出受影响模块、关键路径或配置面。
3. 明确指出高风险点、未验证点和建议重点查看的地方。
4. 检查你的结论是否和实际改动、自检结果一致。
5. 最终把交接信息压缩成下游可直接消费的摘要，而不是过程流水账。

## 输出要求

使用这个 skill 后，你的正式输出应尽量做到：

- 改动摘要具体，能让下游快速定位变化点
- 影响范围清楚，能区分直接改动和潜在受影响区域
- 风险点与测试建议具体，不写泛泛“请回归相关功能”
- 未验证项单独列出，避免混在通过结论里
- 语言面向交接，而不是面向自我复盘

## 常见陷阱

- 把开发过程写得很长，却没有说清最后改了什么
- 只给出文件名，不说明行为层面的影响
- 风险和未验证项写得过轻，导致下游低估问题
- 输出像提交记录而不是交接说明
- 为了显得完整而夸大已验证范围

## 常见自我合理化

- “代码能跑就够了，没必要再写风险点。”
- “下游自己会看 diff，不需要我总结改动。”
- “未验证项就不提了，免得显得这轮不完整。”
- “文件名已经写了，影响范围大家自然能看懂。”

## 完成前验证

- 我是否已经说清本轮真正改了什么，而不是只描述过程
- 我是否写明了影响范围、关键风险和未验证项
- 我给下游的建议重点是否具体到行为或区域，而不是泛泛一句回归
- 我的交接结论是否与实际改动和自检结果一致
