# Quality Review

## 适用场景

当你遇到以下情况时，可按需使用这个 skill：

- 你需要判断当前交付是否足以进入 `Human Gate`
- 你要从需求、实现、测试三方面做综合复核
- 你希望把验收结论拆成多个维度，而不是一句笼统评价
- 你需要形成“通过 / 不通过”的可解释结论

## 何时不要用

以下情况通常不需要使用这个 skill：

- 当前只是做某个局部测试，而不是验收复核
- 材料还远未齐备，连基础判断都无法开展
- 你不承担交付复核职责

## 推荐流程

1. 先确认原始目标、范围和验收标准。
2. 分别检查需求满足度、测试充分性和残余风险。
3. 判断证据是否足以支持交付，而不是只看是否“看起来做了很多工作”。
4. 把可接受风险和不可接受风险区分开写。
5. 输出明确结论，并说明为什么能通过或为什么必须回流。

## 输出要求

- 结论至少覆盖需求满足度、验证充分性、残余风险三条主线
- `approved` / `rejected` 必须与正文证据一致
- 回流建议应能指导后续动作，而不是泛泛一句“再完善”
- 缺失材料、模糊标准和验证空洞都要计入结论

## 常见陷阱

- 只看测试结果，不回看需求是否真的满足
- 只看功能是否可用，不评估残余风险是否可接受
- 把“还没发现问题”误写成“已经足够证明没问题”
- 给出否决结论，却没有可执行回流方向

## 常见自我合理化

- “测试跑通了，需求应该也差不多满足了。”
- “没有明显 blocker，就默认可以送审。”
- “这类风险大概率不会发生，不必写进结论。”
- “验收意见不需要太具体，下游自己理解就好。”

## 完成前验证

- 我是否分别评估了需求满足度、测试充分性和残余风险
- 结论是否由现有材料和证据支撑
- 如果不建议进入 `Human Gate`，我是否写清了为什么以及怎么回流
- 我是否避免把“暂未发现问题”写成“已充分证明无问题”
