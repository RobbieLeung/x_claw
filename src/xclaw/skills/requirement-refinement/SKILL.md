# Requirement Refinement

## 适用场景

当你遇到以下情况时，可按需使用这个 skill：

- 用户原话过于简短，难以直接变成可执行任务
- 任务目标不够具体，容易出现多种解释
- 需要先澄清范围、非目标、限制条件或验收标准
- 你要收敛 `plan` 中的目标、范围、非目标、验收标准，或支持方案调研
- 你要判断当前材料是否足以支持继续规划、实现、测试或验收

## 何时不要用

以下情况通常不需要使用这个 skill：

- 当前需求边界已经很清楚，输入材料足够直接执行
- 你当前只需回答一个很窄、很具体的事实性问题
- 本轮只是在已有明确 spec 基础上做小幅回流，不需要重新定义任务

## 推荐流程

1. 先明确“这次真正要解决的核心问题是什么”，避免把次要问题混进主目标。
2. 写清目标、范围、非目标，区分“本轮必须做”和“明确不做”。
3. 明确约束、风险、依赖和未知项，说明它们会影响什么后续判断。
4. 把模糊愿望改写成可判定的验收标准。
5. 输出下游可直接消费的要求，而不是保留用户原话拼接版。

## 输出要求

使用这个 skill 后，你的正式输出应尽量做到：

- 目标明确，不写成模糊愿景
- 范围和非目标清楚，不把不做的内容留成隐含前提
- 约束、风险和未知项可见，不只写空泛提醒
- 验收标准可执行，便于下游角色据此计划、实现、测试或复核
- 若仍有未知项，要明确哪些结论是事实，哪些只是暂时判断

## 常见陷阱

- 把用户原话简单改写后就当成正式 requirement
- 只写目标，不写范围和非目标，导致下游越界执行
- 把历史问题、未来优化和当前交付混成一次任务
- 验收标准停留在“更好”“支持一下”之类模糊表述
- 把不确定项藏起来，直到后续阶段才暴露

## 常见自我合理化

- “用户意思大概懂了，没必要再结构化。”
- “先开干，边做边理解需求就行。”
- “非目标先不写，下游自然知道做到哪。”
- “验收标准先不细化，做完差不多就行。”

## 完成前验证

- 我是否已经写清目标、范围、非目标，而不是只换一种方式重复原话
- 我是否明确列出了关键约束、风险和未知项
- 我是否给出了可执行的验收标准，而不是笼统愿景
- 下游角色是否可以直接据此计划、实现、验证或复核
